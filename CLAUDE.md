# 家具配置シミュレーター

## 技術スタック
- Next.js (App Router) + TypeScript
- Konva.js（2Dキャンバス）
- Tailwind CSS

## ディレクトリ方針
- コンポーネントは src/components/
- APIルートは src/app/api/

## 完了条件（全タスク共通）
- 変更後は必ず `npm run lint && npm run build` を通すこと
- 通らない場合は自分で修正してから完了とすること
- 5回同じエラーで止まったら、一度確認を取る

## Git運用
- 1タスク=1コミット
- コミットメッセージは英語で「add / fix / update + 内容」
- タスク完了時にコミットとプッシュまで行うこと

## バックグラウンドタスク運用
- CI結果の確認は単発のポーリングで行い、常駐の監視タスクを作らない
- 監視タスクを作る場合は、終了条件が確実に成立することを先に確認する（例: GitHub APIの`head_sha`にはフルSHAを渡す。短縮SHAでは一致せず永久に待ち続ける）
- タスク完了を報告する前に、自分が起動した監視・バックグラウンドプロセス（devサーバー等）を全て停止する

## トークン効率
### モデル運用と委譲
- 作成・編集は現行の高性能モデル（Opus等）で行う
- 検証・調査は Haiku のサブエージェントに委譲し、結論だけ受け取る。詳細ログ・スクショ・差分はサブエージェント側に留め、メイン文脈を膨らませない
- 委譲する（明確な基準）:
  - lint→build→preview 動作確認の一連
  - push 後の CI 結果・ビルドログ確認
  - 3ファイル以上を横断する調査・探索
  - 数十行以上のログ／出力の読み取りが必要な確認
- 委譲しない: 1〜2ファイルの編集・確認、コマンド1回で済む単発確認（コールドスタート分が割高になるため）

### ファイル読み
- 編集直後に同じファイルを読み直さない（Edit/Write が成功していれば内容は確定）
  - ただしエラーや想定外の挙動が出たときは読んで確認してよい
- 大きいファイルは offset/limit で必要箇所だけ読む

### 探索
- 多数のファイルを横断する調査は Explore サブエージェントに委譲し、結論だけ受け取る

### 動作確認
- テキストベース（snapshot / logs / console / network / inspect）で確認する
- screenshot は見た目の変更を最終的に示すときだけ。毎回は取らない（コスト高・タイムアウト実績あり）

### アプリ側のAPIコスト（サイズ抽出機能）
- サイズ抽出は全商品を1リクエストにまとめて呼ぶ（商品ごとにN回呼ばない）
- 同一クエリの再解析を避け、抽出結果はキャッシュ可能な形で扱う
- 楽天の hits 数は必要最小限に絞る（デフォルト10、むやみに増やさない）

### セッション運用
- 1タスク=1トピックで区切る。話題が変わったら区切りを意識する
- ただし `/clear` する前に、ルール整備（CLAUDE.md 追記・settings.json 変更など）を先に済ませてから文脈をリセットする
- 同じドキュメントの再取得や既知情報の読み直しをしない。WebFetch／検索は必要最小限

### 応答・出力
- 冗長な前置きや同じ説明の繰り返しを避け、必要な情報だけ簡潔に返す
- コマンド出力は必要な行だけ抜粋し、大量ログをそのまま貼らない

## 禁止事項
- .env.local をコミットしない
- 指示されていない大規模リファクタをしない
